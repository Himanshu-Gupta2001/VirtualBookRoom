Mine Story 

<POC> End-to-end solution to call Centralized Form Processor backend CloudPage (GEE-35446/GEE-42812)

Description

As the data management lead, I want the backend processing for form submissions to take place on a new CloudPage (main Centralized Form Processor will only collect form data); so that we can improve processing efficiencies and reduce the time it takes to users receive a successful submission confirmation; therefore, improve the overall user experience.

See linked stories for reference.

Acceptance Criteria


SCENARIO 1: Collect web form data
GIVEN the KPMG website calls the Internal Centralized Form Processor CloudPage when a form is submitted
WHEN a user submits a form on the KPMG website
THEN parameter validation will be performed
AND the data will be recorded in a temporary data extension if there are no issues with the validation
AND the GUID will be returned back to AEM
BUT the data will not be recorded in the temporary data extension and no GUID will be returned if any of the parameter validation fails

SCENARIO 2: Backend processing of form data
GIVEN a new CloudPage will be created to for backend processing of form data beyond the point of collection
WHEN a user submits a form on the KPMG website
AND a call is made on this new CloudPage
THEN a new CloudPage will handle the backend processing after the form data is captured, including all existing functionalities such as:***
-Salesforce integration for contact/lead identification, lead creation, Consent Status
-Inquiry type handling for native forms
-Preference management
-All Subscriber updates
-API and Journey Processing
-Error Handling




AEM Story

Centralized Form Processor Optimization *Pre-requirements*

Summary
To cut down on form submission time, Cuc, Bill and Team have worked through a request to split up form submissions into two CFP requests:

We are working on enhancements for Bill to optimize the Centralized Form Processor. The original architecture was to have two separate CloudPages, one to capture the data and another for backend processing. However, we ran into an issue where we couldn’t make the async call between the two CloudPages.

 

We have an idea that we’d like to explore with your team, which involves making two calls to SFMC CloudPages instead of one:

 

-1st call to a CloudPage so the data is captured in a DE, and the user gets an immediate confirmation (currently there is a 20 sec delay)

-2nd call to a CloudPage so kick off the necessary backend processing

 

We’ll provide more details on the call, but please reach out if you have any questions in the meantime.
